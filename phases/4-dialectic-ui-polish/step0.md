# Step 0: design-docs

## 읽어야 할 파일

먼저 아래 파일들을 읽고 프로젝트의 아키텍처와 설계 의도를 파악하라:

- `/docs/UI_GUIDE.md` — **이 step에서 수정할 유일한 파일**
- `/docs/ADR.md` — 특히 ADR-008(순차 논증), ADR-009(차트 라이브러리 없이 인라인 SVG)
- `/docs/PRD.md` — 리포트 5단계 서사 · Split View 규격
- `/docs/ARCHITECTURE.md`

읽기만 하고 수정하지는 말 것 (참고용):
- `web/src/components/report/DialecticSplit.tsx`
- `web/src/components/report/RiskRadar.tsx`
- `web/src/components/ui/Card.tsx`
- `web/src/lib/richText.tsx`

## 배경

리포트 화면의 正/反 대립 뷰가 "의도된 디자인 차이"가 아니라 **통일이 안 된 상태**로 렌더링되고 있다. 실제 원인 4가지:

1. **正/反 골격 불일치** — `DialecticSplit.tsx`의 正 리드는 `<Card>`(4면 테두리, `rounded-md`, `p-6`)인데 反 리드는 `<div className="border-l-2 border-neutral-300 bg-neutral-50 p-4">`로 카드가 아니다. 각진 모서리에 padding도 8px 좁다.
2. **레이더 좌측 치우침** — `RiskRadar`의 고정폭 SVG가 460px 컬럼 왼쪽에 붙는다.
3. **번호 목록 가독성** — 볼드 라벨과 본문이 한 흐름에 이어붙어 통짜 문단이 된다.
4. **테두리와 텍스트가 붙음** — 근본 원인은 **UI_GUIDE가 Card의 `p-6`만 규정하고 좌측 레일 인용/콜아웃 블록의 내부 여백 규격을 정하지 않은 것**이다. 그래서 `p-4`, `pl-3`(상하 여백 0), `pl-4`(상하 여백 0)이 컴포넌트마다 제각각이다.

`docs/UI_GUIDE.md`는 harness `scripts/execute.py`가 **매 step 프롬프트에 가드레일로 주입**하는 문서다. 문서를 먼저 고치지 않으면 이후 step의 에이전트가 옛 규격을 근거로 변경을 되돌린다. 그래서 이 step은 **문서만 고치고 코드는 한 줄도 건드리지 않는다** (phase `2-dialectic-report`의 step0 `design-docs`와 같은 패턴).

## 작업

`docs/UI_GUIDE.md`를 아래 5가지로 갱신하라. 기존 문서의 톤(간결한 규격표 + 왜 그런지 한 줄)을 유지하고, 기존 섹션 구조를 크게 흔들지 마라.

### 1. 정반합 카드 규격 신설 — "미러 액센트 레일"

`## 컴포넌트` 아래에 `### 정반합 카드 (正/反)` 절을 새로 만들고 다음을 규정하라:

- 正/反 카드의 **골격은 완전히 동일**하다: `rounded-md border border-neutral-200 bg-white p-6`. 한쪽만 카드가 아니거나 padding이 다르면 안 된다.
- 주장이 서로 반대라는 것은 **액센트 레일의 방향**으로만 표현한다:
  - 正 = **왼쪽** 레일 2px, 무채색(`neutral-900`)
  - 反 = **오른쪽** 레일 2px, 해당 항목의 severity 색(fatal → red-600 / major → amber-600 / minor → gray-500)
- 두 레일이 가운데 거터를 사이에 두고 마주 보며 정면 대치를 만든다. 이것이 좌우 컬럼의 정체성 신호이므로, 컬럼 헤더가 화면 밖으로 스크롤돼도 어느 쪽 주장인지 알 수 있다.
- 正에 색을 쓰지 않는 이유를 한 줄로 적어라: 낙관 주장에는 severity가 없다. 색은 데이터 의미에만 쓴다(원칙 3).
- 좌우 카드는 모두 **"제목 → 메타 → 근거"** 순서를 지킨다. 反의 `RiskScoreBadge`는 제목 **아래** 메타 줄에 놓는다. 제목 위에 두면 좌우 카드의 첫 줄 baseline이 어긋난다.
- 레일 색 등 정확한 Tailwind 클래스는 `Card` 컴포넌트 내부 구현으로 격리하고, 호출부는 의미(side/tone)만 넘긴다 — `Badge`가 tone→색을 격리한 것과 같은 규율.

### 2. 테두리 블록의 내부 여백 규격 신설

`## 컴포넌트`의 카드 규격 근처에 **"테두리가 있는 블록은 내부 여백을 반드시 확보한다"**는 규격표를 만들어라. 현재 문서에는 Card의 `p-6`밖에 없어서 나머지가 제각각이 됐다.

| 블록 종류 | 규격 |
|---|---|
| 카드 | `p-6` |
| 레일 + 배경 콜아웃 (예: 反의 소결론, 구버전 안내 배너) | 카드와 같은 컨테이너를 쓴다 (`Card` + accent + `bg-neutral-50`). 별도 골격을 손으로 만들지 마라 |
| 레일 인용 (배경 없음, 예: 커뮤니티 목소리·반박 대상 인용) | `border-l-2 border-neutral-300 py-1 pl-4` — **상하 여백을 0으로 두지 마라.** 텍스트가 레일 끝에 딱 붙어 잘린 것처럼 보인다 |

예외로 명시할 것: 진행 화면(`ProgressView`)의 스텝 리스트 아이템은 조밀함이 목적이므로 `p-4`를 허용한다. 목차 네비(`SectionNav`)의 활성 표시와 비교 뷰(`CompareMatrix`)의 컬럼 헤더는 콜아웃이 아니라 표시자이므로 이 규격의 대상이 아니다.

### 3. 번호 목록 규격 갱신 (현재 `## 타이포그래피` 표의 "번호 목록" 행)

현재: `list-decimal pl-6 space-y-3 marker:text-neutral-500 marker:font-medium`

다음으로 바꾸고 이유를 적어라:
- 항목 간격을 `space-y-3` → **`space-y-5`**. 본문 줄간격이 1.8이라 `space-y-3`으로는 항목 경계가 안 보인다.
- 에이전트 산출물의 번호 목록은 `N. **라벨:** 본문…` 꼴이다. **볼드 라벨을 별도 줄로 띄우고 본문은 그 아래 블록으로 내린다.** 라벨과 본문이 한 줄에 이어붙으면 1·2·3이 있어도 스캔이 안 된다.
- 내어쓰기는 `list-decimal`의 기본 동작(marker outside)이 이미 처리하므로 별도 지시가 필요 없다.
- 이 처리는 **번호 목록에만** 적용한다. 불릿 목록은 항목 전체가 볼드인 경우가 많아 분리하면 본문 없는 라벨만 남는다.

### 4. RiskRadar 규격 갱신 (현재 `### RiskRadar` 절)

다음을 추가하라:
- 레이더는 `<figure>` + `<figcaption>`(캡션: "축별 최고 위험도")으로 감싸고, **SVG를 컨테이너 안에서 가로 중앙 정렬**한다. 고정폭 SVG를 블록 컨테이너에 그냥 넣으면 좌측에 붙어 컬럼 안에서 치우쳐 보인다.
- figure는 카드 골격(`rounded-md border border-neutral-200 p-6`)을 쓴다. 테두리 없이 두면 그림이 허공에 떠 보인다.

### 5. 중앙 정렬 금지 규칙에 예외 추가 (현재 `## 레이아웃`의 "정렬: 좌측 정렬 기본. 중앙 정렬 금지(빈 상태 안내 제외)")

이 문장이 4번과 정면 충돌한다. 다음처럼 고쳐라:

> 정렬: 좌측 정렬 기본. **텍스트의** 중앙 정렬 금지(빈 상태 안내 제외). 단, 차트·다이어그램 같은 **도형(figure)은 자기 컨테이너 안에서 가로 중앙 정렬**한다 — 고정폭 도형을 좌측에 붙이면 컬럼 안에서 치우쳐 보인다.

## Acceptance Criteria

이 step은 문서만 바꾸므로 코드 AC가 없다. 대신 아래를 실행해 **아무것도 깨지지 않았음**을 확인하라:

```bash
git diff --stat        # docs/UI_GUIDE.md 한 파일만 변경되어야 한다
npm run build -w web   # 여전히 통과 (코드 무변경이므로 당연)
npm run test  -w web   # 여전히 통과
```

## 검증 절차

1. 위 AC 커맨드를 실행한다.
2. `git diff --stat`의 출력에 `docs/UI_GUIDE.md` 외의 파일이 있으면 **실패**다. 되돌려라.
3. 갱신된 UI_GUIDE를 다시 읽고 자문하라: "이 문서만 읽은 사람이 step 1~5의 변경을 되돌리려 할 여지가 남아 있는가?" 특히 "중앙 정렬 금지"와 "차트 figure 중앙 정렬"이 여전히 충돌하는지 확인하라.
4. 결과에 따라 `phases/4-dialectic-ui-polish/index.json`의 step 0을 업데이트한다:
   - 성공 → `"status": "completed"`, `"summary": "산출물 한 줄 요약"`
   - 수정 3회 시도 후에도 실패 → `"status": "error"`, `"error_message": "구체적 에러 내용"`
   - 사용자 개입 필요 → `"status": "blocked"`, `"blocked_reason": "구체적 사유"` 후 즉시 중단

## 금지사항

- **코드 파일(`web/`, `src/`)을 한 줄도 수정하지 마라.** 이유: 이 step의 목적은 이후 step들이 참조할 가드레일을 먼저 세우는 것이다. 코드를 같이 고치면 문서와 코드 중 무엇이 기준인지 불분명해지고, 커밋이 섞여 되돌리기 어려워진다.
- `docs/ADR.md`·`docs/PRD.md`·`docs/ARCHITECTURE.md`를 수정하지 마라. 이유: 이번 변경은 시각 규격 조정이지 기술 결정이나 제품 요구사항의 변경이 아니다. ADR-009(차트 라이브러리 금지)와 ADR-008(결론 후치)은 그대로 유효하다.
- UI_GUIDE의 `## AI 슬롭 안티패턴` 표에서 항목을 **삭제하지 마라.** 이유: 미러 레일·figure 카드는 그 표의 어떤 항목과도 충돌하지 않는다. 충돌한다고 느낀다면 설계를 잘못 이해한 것이다.
- 새 색상 hex를 도입하지 마라. severity 팔레트(red-600 / amber-600 / gray-500)와 무채색만 쓴다.
- 기존 테스트를 깨뜨리지 마라.
