# Step 0: design-docs

## 읽어야 할 파일

먼저 아래 문서를 읽고 프로젝트의 설계 의도와 ADR 서술 방식을 파악하라:

- `/docs/ADR.md` — 전체를 읽어라. 특히 **ADR-013**(출처는 사실이다 — "형식이 실패한 것이지 검색이 실패한 게 아니다"라는 논증 구조를 이번 ADR이 그대로 재사용한다)과 **ADR-014**(저장소는 바이트를, zod는 의미를 — `artifacts` 테이블의 의미론)를 정독하라.
- `/docs/ARCHITECTURE.md` — 디렉토리 구조, DB 스키마, 패턴 목록, 데이터 흐름
- `/CLAUDE.md` — CRITICAL 규칙과 환경변수 목록
- `src/services/gemini.ts` — 이번 phase가 바꿀 핵심 파일. 현재 `config`에 `thinkingConfig`가 없고 `usageMetadata`를 읽지 않는다는 사실을 눈으로 확인하라.

**이 step은 코드를 한 줄도 바꾸지 않는다.** 문서만 쓴다.

## 배경 — 이 phase가 존재하는 이유

Gemini API 비용이 예상보다 높다. 그런데 **코드에 `usageMetadata` 수집이 단 한 줄도 없다**(레포 전체 grep 0건). 즉 비용이 어디서 나는지 알 수 있는 수단이 현재 존재하지 않는다.

코드와 `data/anvil.db`(9개 run)를 실측해 확인한 사실:

| 사실 | 근거 |
|---|---|
| `thinkingConfig`가 코드에 없다 → `gemini-2.5-flash`는 **동적 thinking이 기본 ON**(콜당 최대 8,192 토큰)이고, **thinking 토큰은 출력 요금($2.50/1M)으로 과금된다**. 제어도 계측도 없다 | `src/services/gemini.ts:269-273`(structured), `:298`(grounded) — 두 `baseConfig` 어디에도 없다 |
| `context` JSON 하나가 하류 4개 콜에 **바이트 동일하게 재전송**된다 — 실측 52,824자로 **하류 프롬프트 입력의 71%** | `thesis.ts:59-62`, `coldCritic.ts:85-87`, `solutionDesigner.ts:71-74`, `verdict.ts:88-92` |
| 그 `context`의 **73.5%가 `communityVoices`(39.8%) + `sources`(33.7%)**. 이 중 `sources`(LLM 자기보고 URL 문자열, 4,449자)는 **正/反/合/판정 어느 에이전트도 논증에 쓰지 않는다** | `data/anvil.db` artifacts 실측 |
| 하류 프롬프트가 `JSON.stringify(x, null, 2)`라 **들여쓰기 공백까지 과금**된다 | 위 4개 파일 |
| YouTube 댓글에 **길이 상한이 없다**(HN은 1,200자 컷이 있다) → 파이프라인 최대 프롬프트의 **상한이 열려 있다** | `src/services/youtube.ts:146-151` vs `src/services/hackerNews.ts:18` |

추정 비용 구조(run당 — **계측으로 확정해야 한다**): thinking ~58% / 출력 JSON ~17% / grounding ~15% / 입력 ~8%.

**핵심 판단**: 통상 비용 최적화의 1순위로 꼽히는 **컨텍스트 캐싱은 입력 8% 슬라이스만 건드린다.** 돈은 thinking에 있다. 그래서 이 phase의 순서는 **계측 → thinking 상한 → 페이로드 다이어트**다.

## 작업

### 1. `docs/ADR.md`에 ADR-016 추가

제목: **ADR-016: 비용을 관측 가능하게 만들고 thinking에 상한을 둔다**

기존 ADR들의 서술 형식(**결정 / 이유 / 기각한 대안 / 트레이드오프 / 하위호환**)을 따라라. 아래 4개 결정을 **근거와 함께** 기록하라. 근거를 빼고 결론만 적지 마라 — 이 레포의 ADR은 전부 "왜 다른 선택지가 아닌가"를 설명한다.

**결정 1 — usage는 `artifacts`가 아니라 새 `usage` 테이블에 넣는다.**
`artifacts`는 "에이전트 산출물"을 의미하며 PK가 `(run_id, kind)`다(ADR-014). usage는 산출물이 아니라 **관측 데이터**이고, **재시도 때문에 한 step에 여러 행**이 생긴다. `kind='usage'`로 우겨넣으면 PK 제약과 의미론이 동시에 깨진다. ADR-014의 "DB는 바이트를 보관하고 의미는 zod가 소유한다"는 **에이전트 산출물**에 대한 규칙이고, usage는 스키마가 고정된 숫자 관측치이므로 컬럼으로 정규화하는 것이 옳다 — 이것은 ADR-014의 예외가 아니라 적용 범위 밖이다.

**결정 2 — 실패한 시도의 usage도 반드시 기록한다.**
검증에 실패한 응답도 과금된다. ADR-013이 "형식이 실패한 것이지 검색이 실패한 게 아니다"라며 실패한 시도의 grounding을 살린 것과 **같은 논리**다: **형식이 실패한 것이지 청구가 실패한 게 아니다.** 성공한 시도만 세면 재시도 비용이 장부에서 사라지는데, 재시도야말로 프롬프트 전문을 재전송하는 가장 비싼 경로다.

**결정 3 — `GeminiService`는 DB를 모른다.**
usage는 `onUsage` 콜백으로 **밖으로 흘려보내고**, DB 기록은 `cli/`가 배선한다. 이유: CLAUDE.md CRITICAL과 ARCHITECTURE "서비스 레이어 격리" — `services/`는 외부 API 래퍼이며 저장소를 모른다. `GeminiService`가 `RunStore`를 import하면 이 계층 규칙이 무너지고, 테스트에서 Gemini를 mock할 때 DB까지 끌려온다.

**결정 4 — thinking은 끄지 않고 상한을 둔다.**
에이전트별 `thinkingBudget`: 기계적 에이전트(`researchPlanner`·`interviewer`)는 **0**, 추론 에이전트는 **2048~4096**. 전부 0으로 끄지 않는 이유: coldCritic의 비판 깊이와 verdict의 판정 품질이 **이 도구의 존재 이유**다(ADR-010이 판정자를 별도 에이전트로 분리한 것도 판정 품질 때문이다). 무제한 dynamic thinking을 없애되 사고 자체는 남긴다.

**기각한 대안** 절에 반드시 넣을 것:

- **컨텍스트 캐싱(explicit `caches.create`) 도입** — 입력 토큰은 전체 비용의 8%뿐이다. 게다가 explicit 캐시는 저장 요금($1.00/1M 토큰/시간)이 따로 붙는데, run 하나가 수 분이면 캐시 수명이 짧아 절감액이 관리 비용을 넘지 못한다.
- **implicit 캐싱을 노린 프롬프트 재배치** — Gemini 2.5는 implicit 캐싱이 기본 ON이고 공통 프리픽스가 2,048토큰을 넘으면 캐시가 히트한다. 그러나 (a) 절감 대상이 8% 슬라이스이고, (b) exact-prefix 매칭이 에이전트별로 다른 `systemInstruction` 때문에 깨지는지 불확실하다. **계측이 붙으면 `cachedContentTokenCount`가 이 질문에 직접 답한다** — 추측으로 프롬프트를 재배치하지 말고, 데이터를 보고 다음 phase에서 결정한다. **이 phase의 계측은 그 데이터를 얻기 위한 것이기도 하다.**
- **모델을 `gemini-2.5-flash-lite`로 내리기** — 출력 단가가 6배 싸지만(=$0.40/1M) 품질 회귀 범위를 지금은 알 수 없다. thinking 상한만으로 절감 목표에 도달하는지 먼저 본다. 계측이 붙으면 에이전트별 비용이 보이므로, 그때 "이 에이전트만 lite로" 같은 판단이 **근거를 갖고** 가능해진다.
- **Batch API(50% 할인)** — 파이프라인은 대화형이고 step이 순차 의존한다. 배치의 지연(수 시간)은 이 도구의 UX와 양립하지 않는다.

**트레이드오프** 절에 넣을 것:
- `usage` 테이블이 run마다 7~20행씩 늘어난다(재시도 포함). 관측의 대가다.
- 단가표가 코드에 하드코딩된다 — Google이 가격을 바꾸면 `src/lib/cost.ts`를 고쳐야 한다. **`cost_usd`는 추정치이지 청구서가 아니다**라는 점을 문서와 코드 주석에 명시한다. 진짜 청구서는 Google Cloud 콘솔에 있다.
- thinking 상한이 품질을 떨어뜨릴 수 있다. 검증은 저장된 run과의 수동 비교로 한다(별도 eval 하네스를 만들지 않는다).

**하위호환** 절에 넣을 것:
- `usage` 테이블은 `IF NOT EXISTS`로 추가되는 **순수 증분**이다. 기존 run은 usage 행이 없을 뿐이고 조회 시 빈 요약이 나온다. 기존 아티팩트·스키마는 하나도 바뀌지 않는다.
- Step 5의 프롬프트 다이어트는 **하류 프롬프트만** 바꾸고 **저장되는 `context` 아티팩트는 그대로 둔다** — 리포트 렌더러와 웹 UI는 영향받지 않는다.

### 2. `docs/ARCHITECTURE.md` 갱신

- **디렉토리 구조**: `lib/` 설명에 `cost` 추가 (예: `cost (모델 단가표 + 토큰→USD 추정 — 순수 함수)`).
- **DB 스키마**: `usage` 테이블 DDL을 추가하고, "도메인 테이블은 이 3개가 전부다"라는 문장을 **정정하라** — 이제 도메인 3개 + 관측 1개다. 관측 테이블임을 명시해 ADR-014의 "산출물을 정규화하지 않는다"와 혼동되지 않게 하라.
- **패턴** 목록에 한 줄 추가: **비용은 관측된다 (ADR-016)** — 모든 Gemini 호출은 재시도까지 포함해 `usage` 테이블에 기록된다. thinking 토큰은 출력 요금으로 과금되므로 별도 컬럼으로 분리해 본다. `GeminiService`는 DB를 모르고 `onUsage` 콜백으로 흘려보낸다.
- **데이터 흐름**: gemini 호출 옆에 usage가 기록된다는 것을 한 줄로 표시하라.

### 3. `CLAUDE.md` 갱신

- **아키텍처 규칙**에 한 줄 추가: 모든 Gemini 호출의 토큰 사용량은 재시도 포함 `usage` 테이블에 기록한다. `services/`는 DB를 모르므로 `onUsage` 콜백으로 흘려보낸다 (ADR-016).
- 이 phase에서 새 환경변수·새 명령어는 **추가되지 않는다.** 없는 것을 적지 마라.

## 금지사항

- **코드를 수정하지 마라.** `src/`·`web/`·`package.json`을 건드리지 마라. 이유: 이 step은 헌법을 쓰는 단계다. 문서와 구현을 한 step에서 하면 구현이 문서를 정당화하는 방향으로 흐른다.
- **과거 ADR을 다시 쓰지 마라.** ADR-013·ADR-014를 인용은 하되 본문을 고치지 마라. 이유: ADR은 시점의 기록이다. ADR-016은 그것들을 **폐기하지 않는다** — 확장할 뿐이다.
- **`sources[]`를 스키마에서 제거한다고 쓰지 마라.** 이유: ADR-013이 `sources`와 `citations`의 상보성을 근거로 **공존**을 명시적으로 결정했고, 리포트 렌더러가 `sources`를 쓴다. 이번에 바뀌는 것은 **하류 프롬프트에 넣지 않는다**는 것뿐이다 — 저장·렌더링은 그대로다. 이 구분을 ADR 본문에서 흐리지 마라.
- **아직 없는 숫자를 사실처럼 쓰지 마라.** 비용 구조(58%/17%/15%/8%)는 **추정치**다. "계측으로 확정한다"는 단서를 반드시 달아라. 이유: 이 ADR의 요지 자체가 "측정하지 않고 최적화하지 말라"이다.
